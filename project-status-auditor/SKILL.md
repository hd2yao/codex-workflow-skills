# Project Status Auditor

用于对一个本地项目做只读状态审计。目标是把“项目现在到哪了、证据在哪里、下一步做什么”讲清楚，而不是临场重写路线图、替代上下文摘要、替代线程迁移或直接改代码。

开始时先说明：`我会用 project-status-auditor 只读梳理项目状态，基于文件、提交、测试和报告证据输出进度、风险和下一步建议。`

## Scope

默认只读执行：

- 不修改代码、配置、文档或生成物。
- 不创建 commit、分支、PR。
- 不读取或输出 `.env`、token、私钥、Cookie、账号密码等敏感内容。
- 如果用户要求顺手修问题，先完成审计，再把修复作为单独任务处理。

## 与摘要/连续性工具的边界

本 skill 和上下文压缩摘要、每日任务摘要、线程接续工具有交集，但职责不同：

| 能力 | 主要来源 | 适合回答 | 不负责 |
|---|---|---|---|
| 上下文压缩摘要 | 当前对话 transcript、最近消息、任务账本 | 这次对话压缩前说了什么、下一轮怎么接 | 系统审计仓库真实进度 |
| 每日任务摘要 | task ledger、work ledger、待确认产物池 | 今天/最近有哪些待办、等待确认、已完成工作 | 判断某个项目是否真的完成 |
| 线程接续/项目归属 | Codex 线程 metadata 和 continuation pack | 换线程继续、fork、预览或调整项目归属 | 对项目文件、提交、测试做证据审计 |
| 项目状态审计 | 本地仓库文件、提交、测试、报告、必要时参考线程摘要 | 某个项目当前阶段、完成度、风险和下一步 | 自动记录所有日常任务 |

使用规则：

- 只想恢复对话、换线程、fork 当前线程、把当前对话关联到项目空间，使用 `codex-thread-bridge`，不要调用本 skill。
- 只想查看今天任务、跨天待办或已完成工作，使用连续性/日报工具，避免扫描项目。
- 只有需要判断“这个项目到底做到哪一步”，并且需要文件、提交、测试或报告证据时，才使用本 skill。
- 可以读取摘要卡片或每日摘要作为线索，但结论必须回到仓库文件、提交、测试、报告或产物证据。

## 触发门槛

不要因为出现“继续项目”“新开对话”“切换项目空间”“接续当前线程”就使用本 skill。那些场景默认交给线程插件处理。

仅在满足以下任一条件时使用：

- 用户明确说“审计项目状态”“盘点真实进度”“验收前检查”“风险/缺口检查”“整理证据和下一步”。
- 当前上下文、日报或线程接续包无法判断项目真实状态，并且继续执行前需要确认仓库证据。
- 用户要求生成正式阶段交接文档，并且交接内容必须包含完成证据、验证记录、风险和后续任务。

如果只是为了给新线程带上下文，优先生成轻量 continuation pack；不要扫描仓库或输出完整审计报告。

## Required Inputs

优先确认：

- 项目路径，默认当前目录。
- 审计目的：阶段交接、继续开发前盘点、风险评估、路线图检查、验收前检查。
- 时间范围，默认最近 30 天提交和主要文档。

信息不足时，先按当前目录和最近提交做合理假设，并在输出中写明。
如果用户只是要切换线程或项目空间，停止本流程并改用线程插件。

## Evidence Sources

按这个顺序收集证据：

1. 项目入口：`README*`、`AGENTS.md`、`PROJECT_STRUCTURE.md`、`docs/`、`plans/`、`reports/`。
2. 当前状态：`git status --short --branch`、最近提交、当前分支与远端关系。
3. 计划和研究：`docs/plans/`、`docs/research/`、`*.discovery.md`、`*.plan.v*.md`、`reviews/`。
4. 验证证据：测试文件、报告文件、benchmark JSON/Markdown、CI 配置、最近生成的 outputs。
5. 业务产物：用户可读报告、导出文件、数据样例、截图或 artifacts。
6. 如果线程工具可用且用户要求跨线程复盘，再读取相关线程摘要；不要把线程记忆当作唯一证据。
7. 如果存在上下文摘要卡片、每日摘要或任务账本，只把它们当作线索来源；关键结论仍需项目证据支撑。

不要只看目录名下结论。每个关键判断至少对应一个文件、提交、测试或报告证据。

## Audit Workflow

1. 建立项目地图：模块、文档、测试、输出目录分别做什么。
2. 找时间线：最近提交和主要计划文档说明了哪些阶段已经完成。
3. 对照目标：把 README/plan/research 中的目标映射到当前实现和产物。
4. 分离状态：
   - 已完成：有代码、文档、测试或运行报告支撑。
   - 进行中：有半成品、TODO、未合并分支或活跃线程。
   - 未开始：只在计划中出现，没有落地产物。
   - 不清楚：缺证据或证据冲突。
5. 检查风险：未提交改动、测试缺口、生成物误入仓库、大文件、凭据风险、计划和实现不一致。
6. 给下一步：输出优先级明确的 5-10 个任务，每个任务写触发条件和验收证据。

## Output Contract

用中文输出，建议结构：

```markdown
**项目状态**
- 当前阶段：
- 一句话结论：
- 证据来源：

**已完成**
| 事项 | 证据 | 备注 |

**进行中 / 未完成**
| 事项 | 当前证据 | 下一步 |

**风险和缺口**
| 优先级 | 风险 | 建议处理 |

**后续安排**
1. 任务：...
   验收证据：...
```

如果用户需要正式交接文档，使用本 skill 内置模板：

```text
assets/project-stage-handoff.md
```

默认写入路径：

```text
docs/handoffs/YYYY-MM-DD-<stage-slug>.md
```

## Common Mistakes

- 把“看起来存在”写成“已经完成”。必须给证据。
- 忽略 dirty worktree。未提交改动会影响审计结论。
- 把生成物、缓存和本地实验数据当作正式交付。
- 只输出摘要，不给下一步可执行任务。
- 因为看到计划文档就默认计划已执行。
