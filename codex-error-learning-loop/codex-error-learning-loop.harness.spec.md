# 周度 Harness 学习闭环 Spec

## 背景和目标

当前系统能在上下文压缩时保留用户要求，也能按需写少量错误学习卡，但没有统一纠错账本、跨线程聚类、候选生命周期和用户可见的周度成长总结。目标是把明确纠正和反复返工先记录为低成本事实，再按上一自然周跨线程分析，形成可持续观察、试验和验证的 Harness Engineering 闭环。

## 范围

- 在线程 `Stop` 和 `PreCompact` 时静默识别明确纠正、方向重置、反复返工和验证失败表达。
- 记录脱敏、可去重的观察事件，不在事件发生时自动修改 Skill、Hook 或规则。
- 每周严格按上一自然周汇总工作、纠错根因和能力沉淀候选。
- 跨独立线程计数；同一线程重复表达不增加独立证据数。
- 持久维护候选状态：`observed -> clustered -> monitoring -> trial -> adopted/rejected -> verified`。
- 对照已有 Skill，优先建议更新现有能力，而不是重复创建。
- 每周一生成并投递独立的用户可见周总结卡片。

## 非目标

- 不读取或保存完整 transcript 正文。
- 不把每次上下文压缩变成一次完整审判。
- 不因单线程普通纠错直接修改全局规则或创建 Skill。
- 不让 Hook 自动执行代码修改、PR 或合并。
- 不把原始日报拼接文件冒充语义周总结。

## 验收标准

- **AC-001**：明确纠正消息会生成一条结构化观察；普通消息不生成。
- **AC-002**：同一线程、同一纠正内容重复触发时幂等去重。
- **AC-003**：观察记录包含线程、项目、时间、问题表现、用户期望、初步类型、证据摘要和状态，并完成敏感信息脱敏。
- **AC-004**：Hook 无用户可见输出，失败不阻塞主流程。
- **AC-005**：周度分析默认范围为上一自然周周一至周日。
- **AC-006**：候选按独立线程计数；至少两个线程才可从 `observed` 进入 `clustered/monitoring`。
- **AC-007**：候选持久保存首次发现、最后发现、周次、线程数、状态、已有能力和下一检查时间。
- **AC-008**：复盘能把已有能力识别为“更新候选”，不会继续建议重复创建同名 Skill。
- **AC-009**：用户周总结展示项目成果、继续事项、跨线程问题根因、成长候选状态、下周动作和无需用户处理项。
- **AC-010**：周总结在周一独立投递到摘要归档线程；日报仍保持简洁。
- **AC-011**：只有跨线程重复、根因可控、已有回归场景且通过治理时，候选才允许进入小范围 `trial`。
- **AC-012**：周度候选后续可标记 `adopted/rejected/verified`，下一周能显示变化而非重复生成同一列表。

## 约束和假设

- 观察层使用确定性启发式，只产生候选事实；根因聚类由周度 Agent 做第一性原理和对抗式审查。
- 周度 Automation 运行在本工作流仓库线程，输出同时投递到摘要归档线程。
- 对 Skill 的任何创建、更新、归档或安装仍须经过 `skill-governance-review`。
