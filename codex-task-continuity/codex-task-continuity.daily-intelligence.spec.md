# Codex 每日连续性增强 Spec

## 目标

每日摘要必须回答三件事：前一天实际做了什么、交付后还需要什么、周期任务是否按计划运行。仓库区只展示已执行动作和真正阻塞，不把原始 Git 事实全部转交给用户判断。

## 验收标准

- **AC-001**：每日自动化读取前一自然日实际活跃的 Codex 任务，并记录为结构化活动；不再只依赖 task/work ledger。
- **AC-002**：活动状态至少区分已完成、已交付待试用、调研完成待实施、进行中、等待用户和阻塞，并展示具体下一步。
- **AC-003**：项目可用 `.codex/continuity.json` 声明周期任务；审计器结合计划时间、调度器、退出码、结构化状态和日志新鲜度判断 `success`、`overdue`、`failed` 或 `unknown`。
- **AC-004**：`ya-fundmind` 的 daily/weekly 被日报识别，daily 显示最近运行日期、业务状态和数据质量，weekly 显示最近证据和下一次计划时间。
- **AC-005**：仓库扫描分别输出相对默认分支和 upstream 的 ahead/behind，不再混用比较基准。
- **AC-006**：高置信已合并分支可自动清理本地 worktree、本地分支和远端分支；有唯一提交的干净分支先 push 保存；分叉、脏改动和冲突停止自动合并。
- **AC-007**：仓库写操作预算从 3 起步，连续 7 次合格运行可按 `3 -> 5 -> 8 -> 10` 提升，风险事件触发回退；只读扫描不受该预算限制。
- **AC-008**：所有自动推断保留证据来源和时间；缺少新鲜证据时不得报告“正常运行”。

## 安全边界

- 周期任务审计只读取声明、调度器状态、日志元数据和白名单 JSON 字段，不读取 `.env` 或业务文件正文。
- 线程活动只存标题、状态、摘要、下一步、项目路径和短证据，不保存完整 transcript。
- 删除分支前必须 fresh fetch/prune、工作区干净、无开放 PR，并有提交可达、内容等价或已合并 PR 证据。
- 动态预算不得超过 10，不因没有处理项而升级。
