# 决策型日报治理与错误学习卡

## 第一性原理结论

- 用户购买日报的“成本”是注意力；日报的产出不是发现数量，而是可以直接理解的项目状态、已完成动作和最小下一步。
- 内部审计报告服务自动化追溯，不服务用户阅读。把报告链接和“证据不足”交给用户等于没有完成摘要工作。
- Git 状态本身可确定：脏改动、ahead/behind、upstream、PR、最后活动和任务关联都应转成明确阶段；只有自动动作实际失败且需要业务选择时才进入用户注意区。
- 15 天无活动是自动收尾优先级；它授权主动推进，但不授权 force、猜测冲突、绕过测试或提交敏感内容。

## Skill 治理结论

- Skill：`codex-task-continuity`
- 结论：继续更新现有全局入口 Skill，不新建 Skill，不增加新的全局 `AGENTS.md` 规则。
- 理由：项目化日报、仓库收尾和任务连续性仍属于同一默认入口；当前问题是该入口的数据和呈现规则不完整。
- 需要修改：Skill、task ledger、repository audit、DailyDigest、测试、现有每日 Automation、全局安装副本和 Obsidian。
- 不需要修改：业务仓库默认规则、GitHub 保护策略、周期任务计划和 YA FundMind follow-up。

## 错误学习

| 字段 | 内容 |
|---|---|
| 触发信号 | 用户指出日报罗列大量“证据不足”和上下文压缩原文，仍需打开内部报告才能理解；项目名不明确，后续安排无法一眼判断。 |
| 原始目标 | 每日用最少注意力看清每个项目昨天做了什么、为什么未合并、今天由谁继续，以及自动收尾实际完成或失败在哪一步。 |
| 错误假设 | 扫描准确等于摘要有用；保留证据路径等于解释充分；不确定就统一写“证据不足”；线程标题可代替项目身份。 |
| 错误类型 | 需求误读、反复返工、流程过轻。 |
| 影响范围 | 每日摘要、仓库收尾、跨项目任务连续性和用户注意力。 |
| 防复发规则 | 用户卡片只展示项目、结果、当前阶段和下一步；内部证据隐藏；仓库必须给明确技术状态；可自动处理先处理；近期继续开发才暂缓并说明对应任务。 |
| 处理状态 | 已修复并完成真实日报验收。 |

## 实现与真实验收

- 扫描器不再生成用户可见“证据不足”，并记录明确阶段、最近活动、15 天优先级、处置方向和下一步。
- 仓库处置账本支持按日期幂等记录；DailyDigest 只展示已处理、近期暂缓和准确失败。
- 安装 Hook 改为优先调用已安装 Skill 的伴随脚本，避免 `~/.codex/hooks` 中旧副本使新规则失效。
- 真实处置完成 Best 2-Hour Build 和 K-Miso 两个仓库；YA FundMind 因今日仍在开发而明确暂缓。
- 最终摘要顶部只显示“本轮仓库处置 3”，不显示原始 54 项内部 finding；历史成果索引与证据路径保留内部，不进入用户卡片。
- 65 项测试、Python 编译和 diff 检查通过；源 Skill、全局安装副本与实际 Hook 一致。
