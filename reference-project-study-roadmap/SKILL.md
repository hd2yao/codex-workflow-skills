# Reference Project Study Roadmap

用于把“参考项目调研”转成可执行的项目路线图。核心原则：先筛选，再理解，再对照当前项目；只借鉴架构、流程、数据源和接口契约，不盲目搬代码。

开始时先说明：`我会用 reference-project-study-roadmap 先筛选高匹配参考对象，再对照当前项目，输出差距分析和按优先级排序的落地路线图。`

## Scope

默认只做研究和计划，不改代码：

- 不复制参考项目代码，除非用户明确要求并确认许可证允许。
- 不把 README 宣传语当成真实能力，必须找代码、文档、配置或运行证据。
- 不引入新依赖、网络调用、数据源或 API 凭据。
- 不读取或输出参考项目或当前项目中的密钥、`.env`、token、Cookie。

## 与 requirement-first-research 的关系

`requirement-first-research` 负责开放式搜索：先提炼真实需求、验证用户线索、覆盖 GitHub/网页/论文/社区/视频等来源，并产出候选清单。

本 skill 通常接在它之后，负责把候选清单压缩成可落地路线图：

- 如果用户还没有参考对象，先用 `requirement-first-research` 搜索和分级。
- 如果用户已经指定少量参考项目，可以直接使用本 skill。
- 搜索结果很多时，不逐个写长报告；先按匹配度、证据等级、维护状态、许可证和当前项目适配度筛到 3-5 个。
- 只对 Top 候选做深度对照；低匹配候选进入“放弃/后续观察”清单。

## Required Inputs

优先收集：

- 参考项目列表：仓库 URL、文档 URL、本地路径或项目名。
- 当前项目路径，默认当前目录。
- 调研目标：架构学习、功能对标、数据源补齐、Agent/Skill 组织、报告方式、测试策略、MVP 差距。
- 期望产物：聊天摘要，或写入 `docs/research/` / 指定目录。

如果用户要求写文件，先遵守当前仓库的目录规则；没有目录时使用：

```text
docs/research/open-source-study.md
docs/research/current-mvp-gap-analysis.md
docs/research/adoption-roadmap.md
```

## Study Workflow

1. 建立参考清单
   为每个参考项目记录 URL、本地路径、许可证状态、主要语言/框架、最近活跃程度和可信证据。

2. 候选筛选
   如果输入来自开放式搜索，先做轻量筛选表：

   ```markdown
   | 候选 | 证据等级 | 匹配度 | 维护/活跃度 | 许可证风险 | 处理 |
   |---|---|---|---|---|---|
   ```

   `处理` 只允许使用：
   - 深度研究：进入逐项目学习，默认最多 3-5 个。
   - 简要参考：只提炼一个可借鉴点，不展开长报告。
   - 放弃：说明不匹配或风险原因。
   - 待验证：证据不足或需要登录/权限。

3. 逐个提炼可借鉴点
   按模块拆解：数据源、任务编排、Agent/Skill 分工、配置、缓存、报告、测试、错误处理、部署方式。

4. 读取当前项目
   先看 README、docs、configs、核心模块、测试、outputs，再用代码搜索确认实现边界。

5. 建立映射表
   固定使用这个表头：

   ```markdown
   | 参考项目 | 可借鉴设计 | 当前项目模块 | 当前状态 | 落地任务 | 优先级 |
   |---|---|---|---|---|---|
   ```

   `当前状态` 只允许使用：
   - 已实现：当前项目有代码或产物证据。
   - 部分实现：概念存在，但缺关键能力或验证。
   - 概念相似：目标相近，但实现路径不同。
   - 未实现：没有明确落地证据。
   - 不适用：业务或约束不匹配。

6. 输出路线图
   给出 5-10 个任务，按影响、风险、依赖和验证成本排序。每个任务必须有验收证据。

## Output Contract

中文输出，建议三段：

```markdown
**参考项目学习**
| 项目 | 核心能力 | 可学习点 | 不建议照搬点 |

**当前 MVP 差距**
| 能力 | 当前证据 | 差距 | 风险 |

**落地路线图**
1. 任务：
   来源：
   改动范围：
   验收证据：
```

写入文件时，三份文档的职责固定：

- `open-source-study.md`：参考项目逐个学习归纳。
- `current-mvp-gap-analysis.md`：参考项目到当前项目的映射和差距。
- `adoption-roadmap.md`：按优先级排序的落地任务。

如果用户只需要一个文档，使用本 skill 内置模板：

```text
assets/reference-project-research.md
```

## Review Checklist

提交或汇报前检查：

- 每个参考项目都有证据来源。
- 没有把许可证不明的代码建议为直接复制。
- 没有把“可借鉴点”写成泛泛的“优化架构”。
- 每个路线图任务都能被验证。
- 当前项目状态没有凭主观印象判断。
- 明确列出不能确定、需要用户确认或需要权限的部分。

## Common Mistakes

- 调研太宽，最后没有当前项目映射。
- 只总结参考项目优点，不说明当前项目是否已经实现。
- 给出任务但没有优先级和验收证据。
- 低估数据源、授权、登录态、API 限制和许可证风险。
- 把研究任务直接升级成开发任务，绕过用户确认。
