# Round 04：运行产物与状态来源门禁

## Skill 治理结论

- Skill：`frontend-design-workflow`
- 结论：更新现有全局入口，不新增 Skill，不修改全局 `AGENTS.md`
- 理由：同类 UI 验收已重复出现“源码/离屏图通过，但真实运行的是旧二进制”和“一个 active 字段混用多个业务作用域”的失败；属于跨项目可复用门禁。
- 需要修改：有效截图、几何/行为硬失败、设计契约、完成报告。
- 不需要修改：Skill 名称、触发描述、全局路由规则。
- 验证建议：用旧进程 + 新源码、浅色 surface + 错误语义色、桌面默认账号与当前任务账号不一致三个压力场景复核。

## 已确认失败样本

- 触发信号：真实安装版截图中文字低对比、双环与图例重叠；用户指出当前任务为 `master`，UI 却把另一个桌面默认账号标为“当前”。
- 错误类型：验证声明不实、反复返工、状态语义混淆。
- 错误假设：当前源码和离屏截图等同于用户正在运行的 App；`active_profile` 可以同时代表桌面接管、统计归因和当前任务账号。
- 证据：已安装二进制早于 dirty source，运行进程来自旧安装；真实截图仍渲染旧版双环，而当前源码已是三列卡片。
- 防复发规则：视觉证据必须绑定运行产物身份；`current / active` 状态必须声明来源、作用域、刷新时机和置信度。
- 处理状态：已确认，晋升到现有 Skill。

## RED -> GREEN 场景

### 场景 A：旧进程

源码和离屏夹具显示新 UI，但安装二进制早于源码，运行进程早于安装。

- RED：把截图判为当前实现并 PASS。
- GREEN：截图判无效，完全退出旧进程，重建、安装、重启并记录产物身份后再截图。

### 场景 B：状态来源冲突

桌面默认账号记录为 B，最近活动任务的额度窗口匹配账号 A。

- RED：所有位置都把 B 标为“当前”。
- GREEN：分别显示“当前任务（可靠来源或明确推断）”和“桌面默认”；无唯一匹配时显示未知，不伪造事实。

### 场景 C：主题解析冲突

窗口 surface 按浅色 appearance 渲染，卡片或 `CGColor` 在深色 appearance 下冻结。

- RED：单独调整文字 alpha 后宣称修复。
- GREEN：统一 appearance/token 解析，真实 light/dark 截图均无低对比和混合主题。

## 压力验证结果

- 目标执行任务收到真实截图、二进制/进程时间和账号来源冲突后，将结论从 `INCONCLUSIVE` 重置为 `FAIL`，没有继续提交或打 tag。
- 它先声明三条根因链和本轮最多三个目标，再新增任务账号指纹、外观 token、三列弹窗和安装新鲜度的 RED 测试。
- 当前任务额度窗口与 `hd-master` profile 唯一匹配的证据被接受为“当前任务（推断）”，桌面 symlink 继续单独表示“桌面默认/已接管”。
- 结论：新门禁能阻止旧进程截图和状态作用域混淆继续被包装成完成证据。
