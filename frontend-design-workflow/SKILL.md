# 前端设计工作流

## 核心原则

参考不是风格词，而是证据。先把参考项目、当前项目和用户目标拆成可执行的设计契约，再写代码；设计契约一旦确认就是 Design Lock，后续迭代不能无理由漂移；截图必须有效、可比、可复核，源码测试、编译通过和“看起来没问题”都不能替代视觉验收。

## 路由

先判断任务类型：

| 场景 | 处理 |
|---|---|
| 小范围样式修复 | 读取目标组件、相关样式和截图，直接修复并做局部视觉验证 |
| 新页面/重构/仿照参考 | 执行完整流程：目标澄清 -> 参考拆解 -> 当前审计 -> 设计契约 -> 实现 -> 截图验收 |
| 用户要求“照着做/尽量一样/严格复刻” | 启用严格复刻模式；参考的信息架构、区域比例、密度和组件层级默认锁定，只允许为产品功能差异做有证据的偏离 |
| 只做评审 | 跳到“验收评审”，输出问题、严重度、修复建议和需要截图的证据 |
| 用户要求多方案探索 | 先做参考拆解和设计契约，再按需使用 `superdesign` 做保真复现与分支方案 |
| 跨轮或中大型 UI 项目 | 读取或建立项目级 `DESIGN.md` 和页面级验收记录，启用 Design Lock |

如果当前任务明显涉及 UI 设计、实现或评审，同时读取并遵守 `frontend-ui-guardrail`。如果需要浏览器验证，按需使用 `playwright` 或可用浏览器工具。若使用 `superdesign`，必须读取并遵守 `superdesign` skill，不要只在文字里说“使用设计工具”。

## 1. 目标和输入

开始前只问会影响方向的问题；用户已经给了目标、参考和范围时不要反复澄清。

固定提取：

```markdown
设计目标:
- 目标界面：
- 用户/场景：
- 主要任务：
- 成功标准：
- 不做范围：
- 参考模式：严格复刻 / 结构借鉴 / 风格探索

输入证据:
- 当前项目：
- 参考项目/截图/Figma：
- 品牌/资产：
- 技术栈/组件库：
- 约束：
```

缺少关键信息时，最多问 1 轮高信号问题：目标页面、参考优先级、允许改动范围、保真还是借鉴、是否必须保留现有组件库。

## 2. 参考拆解

当用户给了开源项目、竞品、截图、Figma 或源码时，先拆参考，再设计。

必须读取或确认：

- README、入口页面、路由、代表组件、全局样式、主题 token、Tailwind/CSS 配置、图标和真实资产。
- 参考页面的实际渲染分支；不要只看文件名、import 名或组件名。
- 如果可运行，启动参考或查看官方截图；否则用源码和公开截图建立证据。
- 许可证和品牌边界；只借鉴结构、节奏、交互和设计系统，不复制无授权代码、资产或独特品牌表达。

严格复刻模式必须先建立“参考基线”，没有基线不得实现：

```markdown
参考基线:
- 仓库/版本/commit：
- 目标页面和真实渲染分支：
- 参考截图及状态：
- 参考窗口/viewport：
- 区域映射：header / overview / tabs / content / footer
- 允许偏离：
- 禁止偏离：
```

必须同时读取目标截图和对应页面源码；只读 README、只看组件名、只抄颜色或只凭记忆都不算完成参考拆解。参考与当前产品功能不同的区域，保留参考的布局角色和视觉权重，再替换业务内容。

输出“视觉 DNA”：

```markdown
参考拆解:
- 信息架构：
- 布局骨架：
- 密度/间距：
- 字体/层级：
- 色彩/token：
- 组件语言：
- 状态/交互：
- 动效：
- 独特记忆点：
- 不能照搬：
```

如果参考只是“看起来高级”，继续拆到具体结构和 token；拆不出来就不能开始实现。

## 3. 当前审计

实现前先理解当前项目：

- 读取目标页面、局部组件、共享布局、UI primitives、全局样式、主题 token、配置和路由。
- 截取当前页面或运行现有故事/预览，确认真实问题。
- 记录与参考的差距：信息缺失、布局层级、密度、组件一致性、颜色/字体、状态、交互、响应式、可访问性。
- 审计几何结构：窗口或 viewport 尺寸、根表面、滚动容器、document/content view、固定宽高、min/max 约束、裁切边界、顶部/底部固定区域。

不要在不了解现有组件库时重造 Button、Card、Input、Tabs、Modal、Toast、Sidebar 等基础组件。

复杂页面必须写出几何模型：

```markdown
几何模型:
- 外层表面：覆盖范围、圆角、背景、裁切
- 滚动层：方向、viewport、document/content 尺寸关系
- 固定层：header / footer / toolbar
- 宽度策略：固定 / 自适应 / 断点；对应 min/default/max
- 高度策略：首屏、内容高度、顶部/底部状态
- 文本策略：最长名称、最大数字、缺失值、缩放/截断/换行
```

固定内容宽度必须同时锁定窗口最小/最大宽度，或改为随 viewport 自适应；不能让内容固定宽度与可任意缩小的窗口并存。滚动内容必须被外层表面裁切，背景、边框和固定 footer 不得随 documentView 一起消失。

## 4. 设计记忆和 Design Lock

UI 工作流分三层记忆，按任务规模选择使用：

| 层级 | 内容 | 何时使用 |
|---|---|---|
| 全局层 | 本 skill 的反 AI 味、参考拆解、截图验收和浏览器验证规则 | 所有 UI 任务 |
| 项目层 | `PRODUCT.md`、`DESIGN.md`、设计系统文档、主题 token、组件目录 | 中大型项目、跨轮 UI、多个页面共享风格 |
| 页面层 | 页面级参考图、`VISUAL_ACCEPTANCE.md`、截图历史、已知问题 | 仿截图/仿参考、复杂页面、反复返工页面 |

如果项目已有 `DESIGN.md` 或同等设计系统文件，先读取并把它作为 Design Lock。若没有，只有在以下情况才建议新建或更新：

- 用户明确希望后续多轮保持同一视觉语言。
- 本次会影响多个页面、设计系统、主题 token 或核心布局。
- 当前任务是从参考项目建立一套可复用视觉方向。

Design Lock 至少写清：

```markdown
Design Lock:
- 视觉方向：
- 产品性格：
- 色彩/token：
- 字体/层级：
- 间距/密度：
- 圆角/阴影/材质：
- 组件语言：
- 动效：
- 禁止漂移：
- 可调整范围：
```

后续实现、修复和 polish 必须说明是否保持 Design Lock。需要偏离时，先更新设计契约或项目 `DESIGN.md`，不要在代码阶段顺手改风格。

## 5. 设计契约

非平凡 UI 改动必须先写短设计契约，再实现。聊天中写即可；跨页面或需要后续复用时再落盘到项目内合适位置。

```markdown
设计契约:
- 任务模型：
- 信息模型：
- 状态来源：每个 `current / active / selected / connected` 字段的权威来源、作用域、刷新时机、置信度和降级文案
- 布局模型：
- 几何模型：
- 风格模型：
- 参考模式：
- 参考映射：
- 组件/token 计划：
- 状态覆盖：loading / empty / error / disabled / active / success
- 响应式策略：
- 验收截图：
- 硬失败条件：溢出 / 裁切 / 重叠 / 参考关键区域缺失 / 无有效对照图
- Design Lock：
```

设计契约必须把“参考拆解”映射到当前项目，说明哪些采用、哪些不采用、为什么。

用户、账号、会话、设备、工作区等状态不得只靠名称推断。`当前任务账号`、`桌面默认账号`、`最近选择账号` 和 `统计归因账号` 必须使用独立字段；无法可靠检测时显示准确的未知或推断状态，不得把一个全局记录同时包装成多种“当前”。

## 6. 设计工具分支

需要多方案、保真复现或复杂重构时，优先用 `superdesign`：

1. 先完成 `.superdesign/init/` 和设计系统上下文。
2. 对现有 UI 先做像素级当前复现作为 ground truth。
3. 再用 branch mode 生成 1-2 个明确方向的变体。
4. 用户确认方向后再实现代码。

不要跳过“当前复现”直接生成新风格；那会让工具根据泛化审美发明页面。

如果用户明确要求使用外部设计 skill：

- `UIUX Pro Max`：适合查风格、色彩、字体、UX、技术栈规则和生成设计系统；不要默认全量安装或让它覆盖项目组件约束。
- `Impeccable`：适合 critique/audit/polish 和反 AI 味审查；不要默认启用其 hooks 或命令体系。
- `Claude Design` 类 skill：适合 HTML artifact、设计画布、原型和多变体探索；产品前端重构仍必须回到当前项目组件和截图验收。
- `StyleSeed` 类 skill：适合 Design Lock、Quality Gate 和反 AI 味规则；默认只吸收思想，不全量搬组件/规则库。
- `Visual Verdict` / Browser QA 类 skill：适合截图评分和真实浏览器验证；默认抽象成工具无关的验收门禁。

## 7. 组件来源层

需要复杂组件时，先找项目内已有实现，再考虑外部组件来源：

1. 项目组件库、Storybook、shadcn/ui wrapper、Radix wrapper、内部 patterns。
2. 参考项目同类组件的结构和交互，不复制无授权代码。
3. 21st.dev Magic、优秀开源 blocks 或设计系统示例等外部组件来源。

外部组件只负责提供实现参考，不负责决定整体视觉风格。引入前必须检查许可证、依赖、可访问性、主题 token 适配和项目代码风格；需要 API key 或外部服务时，先说明，不默认安装。

## 8. 实现

执行规则：

- 先实现信息和布局，再加风格和动效。
- 优先复用项目组件、设计 token、CSS 变量和图标库。
- 需要新 token 时用语义命名，避免散落 raw hex、任意阴影和任意圆角。
- 工具型/仪表盘/设置页保持工作型密度，不做营销式 hero 和装饰卡片堆叠。
- 文案按用户任务命名，不写系统内部实现词。
- 固定格式 UI 元素要给稳定尺寸、响应式约束和长文本策略。
- 外层表面、滚动内容和固定 footer/header 分层实现；滚动到顶部或底部时，内容不能越过圆角、标题栏或窗口边界。
- 原生窗口明确 min/default/max size；Web 明确断点和容器宽度。每个固定宽高都要有缩小、增长和数据刷新时的行为。
- 动效必须服务状态变化，尊重 reduced motion，不遮挡或延迟核心信息。
- 中大型页面不要一次性重写完。按区域迭代：主画布/核心内容 -> 关键组件 -> 状态与交互 -> 辅助面板 -> 整体整合。
- 每次迭代只修一组视觉问题，保留可回滚 diff。

常见拒绝模式：

- 只换渐变、圆角和阴影。
- 模仿参考的表面配色，却没有复刻信息密度、布局节奏和组件状态。
- 使用无授权 logo、截图或产品图片。
- 生成 SVG 涂鸦替代真实产品图。
- 在截图失败后用文字解释，而不是修布局。

## 9. 验收评审

完成前做真实可见结果验证。Web 默认启动本地预览并用浏览器/Playwright 检查桌面和移动视口；原生或桌面 UI 用对应截图工具。

### 有效截图证据

截图必须满足：

- 目标应用单独可辨，无遮挡弹窗、无其他窗口文字穿透；透明材质应放在干净、可控背景上复验。
- 包含完整外框或明确标注的裁切区域；滚动页面同时提供顶部和底部状态。
- 参考图和成品图使用相同状态、相近比例或统一裁切，并排或逐区对照。
- 文件名或报告标明页面、窗口尺寸、状态和轮次，不能把旧截图当作新实现证据。
- 截图来自本轮目标运行产物，而不是旧安装包、旧进程、缓存页面、storybook mock 或只包含同名组件的离屏夹具。

原生 App、桌面工具和需要 build/install 的前端在截图前必须记录运行产物身份：源码 commit 或 dirty fingerprint、构建时间/版本、安装路径、二进制时间或 hash、进程 PID/启动时间。若进程早于本轮安装、二进制早于目标源码、版本相同但产物身份无法区分，先完全退出旧进程并重建重启；否则截图无效。Web 至少记录当前 dev server 启动时间、代码 revision 和目标 URL，避免命中旧服务或缓存。

任一条件不满足，结论只能是 `INCONCLUSIVE`，不能因为“截图已保存”而 PASS。

### 几何门禁

最小检查：

- 控制台无关键错误，页面可加载。
- 目标任务能一眼理解，主操作明确。
- 参考映射中的关键布局、密度、组件语言已经体现在当前界面。
- 必须信息、状态和边界文本可见。
- Web 在 375px、768px、1440px 至少三个宽度检查；原生 UI 在最小、默认、最大或较宽窗口检查。
- 除明确设计的横向画布外，不得有横向滚动；任何子视图不得越过根表面、窗口圆角、标题栏或 footer。
- 滚动到顶部和底部时，背景、边框、固定区域连续，内容不从外框上下穿出。
- 最长名称、最大数字、0 值、缺失值、loading/error、中文英文混排不重叠、不裁切、不推动关键控件跳位。
- 键盘焦点、图标标签、颜色对比、触控尺寸满足基本可访问性。
- light/dark/high contrast 或项目已有主题没有明显破坏。
- 同一窗口只能使用一套已解析 appearance/token；不得混用浅色 surface、深色 card 和错误 appearance 下冻结的语义文字。动态系统色转静态绘制色或 `CGColor` 时必须在目标 appearance 下解析并验证。
- 标为“当前/活跃/已连接”的业务状态与其权威来源一致；状态来源不一致、作用域混淆或把推断值当事实，按行为错误处理。

源码断言、组件名检查、单元测试和编译只证明结构或行为，不证明像素、裁切、溢出和参考一致度；视觉门禁必须以已确认身份的新鲜运行产物为准。

### 视觉评分门禁

有参考截图、Figma、竞品页面或明确视觉目标时，必须做截图评分。没有参考图时，用 Design Lock 和当前页面验收清单作为参考基准。

```markdown
Visual Verdict:
- 证据：参考图 / 成品图 / 窗口尺寸 / 状态 / 轮次
- 布局准确度：?/100
- 字体/层级：?/100
- 色彩/token：?/100
- 响应式：?/100
- 交互状态：?/100
- 内容完整度：?/100
- 参考一致度：?/100
- 反 AI 味：?/100
- 总分：?/100
- 结论：PASS / REVISE / FAIL / INCONCLUSIVE
```

默认阈值：

| 结论 | 分数 | 动作 |
|---|---:|---|
| PASS | >= 90 | 可交付 |
| REVISE | 70-89 | 修复 P0/P1 后复验 |
| FAIL | < 70 | 回到设计契约或参考拆解，不要继续表面 polish |
| INCONCLUSIVE | 无有效证据 | 重新生成可比截图，不得交付 |

评分是证据摘要，不是意图清单。如果用户要求严格复刻，参考一致度和总分都必须达到 95，并逐区列出仍有差异；没有参考与成品的有效对照图时直接 `INCONCLUSIVE`。任何文字重叠、内容越界、窗口外框被滚动内容穿透、关键区域缺失、运行产物身份不明、关键“当前/活跃”状态来源错误都直接 `FAIL`，不允许用其他高分抵消。若没有参考图，PASS 可基于 Design Lock 和验收清单判断，但必须说明“无参考图，评分基于设计契约”。

### 综合评分

无论是否有参考图，都用两层评分：

```markdown
设计健康:
- 洞察到行动：?/4
- 工艺质量：?/4
- 可信构建：?/4

技术健康:
- 可访问性：?/4
- 响应式：?/4
- 主题/token：?/4
- 性能/动效：?/4
- 反 AI 味：?/4
```

任一项低于 3，先返工最高影响问题再交付。不要把失败截图包装成“后续优化”。

## 10. Fix 循环

截图评分不通过时，进入 Fix 循环：

```text
截图/报告 -> P0/P1/P2 排序 -> 修改最小区域 -> 重新截图 -> 重新评分
```

规则：

- 每轮最多修 3 个最高影响问题。
- 最多 5 轮；5 轮仍未 PASS 时停止，报告最佳分数、反复失败项和需要用户/设计确认的问题。
- Fixer 只读设计契约、Design Lock、Visual Verdict 和相关代码；不要重新发明风格。
- 若连续两轮总分提升 < 5 分，回到参考拆解或设计契约，而不是继续调 CSS。
- 视觉 PASS 前不得发布、打 tag、做正式版本 bump 或宣称完成；为生成运行截图所需的本地构建/安装不算发布。

以下说法都不是完成证据：

- “结构已经对了”“大体接近”“编译和测试都通过”。
- “截图存在”“能看到目标模块”“透明感已经保留”。
- 只描述实现了哪些组件，却不指出参考与成品的可见差异。

## 输出

实现前：

```markdown
设计目标:
设计记忆:
参考拆解:
当前审计:
设计契约:
验收计划:
```

完成后：

```markdown
变更:
```
