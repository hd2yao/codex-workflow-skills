# 需求优先调研

用这个 skill 处理开放式调研：先理解用户真正要解决的问题，再把用户给出的例子当作待验证线索，而不是搜索边界。

核心原则：需求优先，示例校验，平台分层，证据分级，最后映射到用户当前流程。

## 使用边界

使用本 skill：

- 用户要求“全网搜索、调研、找相关项目、看看有没有现成方案”。
- 用户给出不确定例子，例如“我印象中有一个 X，但不确定是否符合”。
- 任务需要覆盖 GitHub、Skill/Agent 目录、官方文档、论文、博客、X/Twitter、Reddit、YouTube、B站或中文社区。
- 输出需要判断“哪些可用、哪些不可用、哪些只能参考、如何放进当前工作流”。

不要单独使用本 skill：

- 用户只要求读取一个已给 URL 或单个仓库；这时先读目标内容。
- 用户只问一个可直接回答的事实；直接查证并回答。
- 用户已经指定只整理某几个参考项目；优先用 `reference-project-study-roadmap`。

如果当前环境有 `agent-reach`，互联网和平台搜索部分同时使用它；本 skill 负责“搜什么和如何判断”，`agent-reach` 负责“用哪个平台后端和命令”。

## 工作流程

1. 拆分需求和线索

   开始搜索前先写出四类信息：

   | 类别 | 说明 |
   |---|---|
   | 真实目标 | 用户最终要做的决策、系统、流程或报告 |
   | 必须满足 | 成功结果不可缺少的能力或约束 |
   | 用户例子 | 用户提到的项目、Skill、名称、平台、人物或概念，全部标为“待验证” |
   | 非目标 | 用户没有要求、暂时不做或不能扩大到的范围 |

   用户例子只能扩大搜索面，不能缩小搜索面。不要因为用户举了一个例子，就只围绕这个例子找。

2. 先验证用户例子

   对每个例子做精确检索和适配判断：

   | 状态 | 含义 |
   |---|---|
   | 符合 | 例子直接覆盖用户核心需求 |
   | 部分符合 | 覆盖一个子能力，但不能独立满足整体需求 |
   | 不符合 | 名称存在但目标、平台、许可证或能力不匹配 |
   | 未验证 | 找不到可靠来源，或平台暂时不可访问 |

   如果用户之后指出一个你没找到的明确项目或 Skill，把它当作搜索策略失败信号：补上精确名称、同义词、平台目录和生态关键词后重新检索，不要只解释“没搜到”。

3. 建立查询矩阵

   至少覆盖这些查询维度：

   | 维度 | 查询内容 |
   |---|---|
   | 问题词 | 用户痛点、失败模式、流程问题、场景词 |
   | 方案词 | 方法论、框架、工具、库、插件、Skill、harness、workflow |
   | 产物词 | repo、template、example、paper、case study、agent skill、MCP、CLI |
   | 平台词 | Codex、Claude Code、Cursor、OpenAI、Anthropic、LangChain、AutoGen 等相关生态 |
   | 用户例子词 | 精确名称、slug、作者名、仓库名、缩写、别名 |
   | 中英文同义词 | 中文需求词和英文行业词都要覆盖 |

   对“自我纠错/自我进化”这类需求，不要只搜 `self improving agent`；同时搜 `reflection`、`agent memory`、`retrospective`、`error taxonomy`、`requirement clarification`、`context pollution`、`learning loop`、`skill optimization`、`harness engineering` 等相邻概念。

4. 分层覆盖平台

   | 来源层 | 重点检查 |
   |---|---|
   | GitHub/代码 | README、issues、discussions、最近提交、license、示例和测试 |
   | Skill/Agent 目录 | Skill Hub、agent skills、插件目录、MCP/工具目录、个人技能仓库 |
   | 官方文档/博客 | 是否有正式概念、API、限制、推荐做法 |
   | 论文/技术报告 | 方法是否真实存在，是否只是研究原型 |
   | Reddit/X/Twitter/V2EX | 社区是否实际使用，常见失败反馈是什么 |
   | YouTube/B站/播客 | 是否有教程、演示或访谈；只当社区证据，除非读过字幕/正文 |
   | 中文网页/论坛 | 中文场景、国内工具、平台限制和替代方案 |

   如果某个平台无法访问、需要登录或搜索能力不足，在报告中写明盲区，不要把“没搜到”说成“不存在”。

5. 给证据分级

   | 等级 | 证据 |
   |---|---|
   | A | 官方文档、源码、release、可运行示例、活跃维护记录 |
   | B | 高质量博客、教程、社区长帖、带证据的经验复盘 |
   | C | 社交媒体短帖、视频标题、二手转述、未读全文的摘要 |

   推荐采用某个方案前，至少要有 A 级证据；只有 B/C 级证据时，只能标为“可继续调研”或“作为概念参考”。

6. 映射到当前流程

   对每个候选项固定回答：

   | 问题 | 说明 |
   |---|---|
   | 能解决哪部分需求 | 对应用户的真实目标，不对应宣传语 |
   | 可直接使用什么 | Skill、脚本、模板、流程、数据结构或报告格式 |
   | 不应该使用什么 | 过重、过窄、平台绑定、许可证不清或维护差的部分 |
   | 对当前流程的影响 | 是否会让流程更重，是否应放入 spec、hook、Obsidian、全局规则或项目规则 |
   | 下一步 | 采用、参考、继续验证、放弃 |

   不要把所有发现都升级成全局规则。只有跨项目高频、默认入口型、会显著减少误判的内容，才考虑进入全局规则；普通经验优先沉淀到 Skill、项目规则或 Obsidian 索引。

## 输出格式

调研报告按这个顺序输出：

1. 结论：推荐采用什么、不采用什么、还需要验证什么。
2. 用户例子校验：用户给出的每个例子是符合、部分符合、不符合还是未验证，并说明原因。
3. 搜索覆盖：列出已覆盖平台、关键词族和盲区。
4. 候选清单：

   | 候选 | 来源 | 证据等级 | 匹配度 | 可借鉴点 | 不建议使用点 |
   |---|---|---|---|---|---|

5. 当前流程映射：哪些放进现有流程，哪些去掉或轻量化，哪些留作后续规划。
6. 后续路线图：按 P0/P1/P2 给出可验证动作。

## 常见错误

- 只搜索用户给出的例子，没有围绕真实需求扩展关键词。
- 把用户的“我印象中”当成事实，而不是待验证线索。
- 只看 README 或宣传文案，不看源码、issue、license、最近活跃度。
- 平台覆盖只停在网页搜索，漏掉 GitHub、Skill 目录、社媒、视频和中文社区。
- 没有区分“可直接使用”和“只适合参考”。
- 把一次调研结果直接写进全局规则，导致后续工作流越来越重。
